以下のテンプレートを `{{PROJECT_NAME}}/docs/design-principles.md` として生成してください。

## 生成条件

常に生成します（全プロジェクト共通）。

## 使い方

Issue/PR の起票時に「この変更はどの原則に基づくか」を明示させる。
CLAUDE.md の設計原則セクションから参照される詳細ドキュメント。

---

```markdown
# 設計原則

プロジェクト全体を貫く原則。Issue/PR の起票時に「この変更はどの原則に基づくか」を明示する。

## 基本原則

| # | 原則 | 要点 | 違反例 |
|---|------|------|--------|
| 1 | **責務の分離** | 一つの知識を複数モジュールが重複して持たない。データの種別（事実/状態/履歴/理由）ごとに所有者を明確にする | 別コンテキストのモジュールに同じデータを二重保存する |
| 2 | **参照はするが書き戻さない** | 他コンテキストのデータはスナップショット取得のみ。元システムへの書き戻し禁止 | モジュールAからモジュールBのステータスを直接更新する |
| 3 | **段階導入と独立動作** | 各モジュールは他モジュールが未導入でも単独で動作する。連携は「あれば便利」レベルに留める | モジュールBがないとモジュールAが起動しない |
| 4 | **ワンファクト・マルチビュー** | データは発生元で1度だけ記録する。各モジュールは同じデータを異なる視座で表示する | エンティティを別コンテキストのストレージにコピーする |
| 5 | **共通識別子による横断** | 全モジュール共通の識別子（案件ID等）を設け、横断参照の基盤とする | モジュール独自の管理番号体系を新設する |

## 並行開発ルール

複数の開発者・エージェントが同時に作業する際の制約。

| ルール | 内容 |
|--------|------|
| **パッケージ所有権** | 1パッケージを同時に2人以上が編集しない |
| **フロント+バック一体** | UI パッケージと対応する API エンドポイントは同一担当が扱う |
| **共通パッケージ変更時の義務** | 共通パッケージ（domain 等）を変更する場合、**変更前に** `docs/constraints/` に制約条件を記録する |
| **型の追加は自由** | 型の変更・削除は制約条件への記録が必須 |
| **共通 UI コンポーネントの props 変更** | 後方互換を保つ。破壊的変更は制約条件として記録する |

## 環境変数による段階切替

外部サービスが未整備でもローカルで開発・テスト・CI が動く設計にする。

```
未設定 = ローカル動作（外部サービス不要）
```

| パターン | 切替内容 | 段階例 |
|----------|---------|--------|
| 認証方式 | モック認証 → 本番認証基盤 | `email` → `sso` |
| DB 実装 | インメモリ → ファイル → 本番 DB | `memory` → `jsonfile` → `production` |
| フロントエンド永続化先 | ローカルストレージ → API | `local` → `api` |

## 段階導入の分解パターン

機能を段階的に導入する際の共通パターン:

```
仕様確定 → ドメイン型定義 → フロントエンドモック → API 実装 → API 接続 → 認証統合 → 結合テスト
```

- **フロントエンドモック**: 外部サービスが揃うまではここで止める設計とする。ローカルストレージ等のモックで単体動作を保証する
- **段階分離**: Step A（外部サービスなしで可能な範囲）と Step B（外部サービス必須の範囲）を明確に分ける

## コミットメッセージ規約

```
{type}({scope}): {summary}
```

| 要素 | 説明 | 例 |
|------|------|-----|
| `type` | 変更種別 | `feat`, `fix`, `docs`, `chore`, `refactor`, `test` |
| `scope` | 影響パッケージ | `domain`, `api`, `web`, `root` |
| `summary` | 変更内容の要約 | 日本語または英語 |

例:
```
feat(domain): エンティティの型定義を追加
fix(api): CORS 設定の修正
docs(constraints): 001-responsibility-split を追加
chore(root): lint 設定の更新
```

## エスカレーション基準

以下に該当する場合、`docs/issues/reports/` に事象を起票してユーザーに確認する:

| 基準 | 例 |
|------|------|
| **仕様書に記載がなく推測での実装が危険** | 承認フローの分岐条件が不明 |
| **設計原則が矛盾する状況** | ワンファクトと参照禁止が衝突する |
| **認証・認可フローの変更** | 認証方式やリダイレクト設定の変更 |
| **既に動作しているコードへの破壊的変更** | domain 型の削除・変更 |
| **外部システムとの連携仕様の不明点** | 外部 API の仕様が未確定 |
| **コスト影響が大きい変更** | DB の設計変更、クラウドリソースの追加 |
```
