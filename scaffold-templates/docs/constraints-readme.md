# 制約条件 README テンプレート

以下のテンプレートを `{{PROJECT_NAME}}/docs/constraints/README.md` として生成してください。

---

```markdown
# 制約条件（Constraints）

実装過程で発見・決定された制約条件を記録するディレクトリ。

## 目的

- 設計判断とその根拠をトレーサブルにする
- 将来の開発者（人間・AI 問わず）が「なぜこうなっているか」を理解できるようにする
- 暫定措置と恒久対応の区別を明確にする

## ファイル命名規約

`{連番3桁}-{kebab-case の概要}.md`

例: `001-repository-di-pattern.md`

## テンプレート

各制約条件ドキュメントには以下のセクションを含める:

```markdown
# 制約条件 NNN: タイトル

## 発生日

YYYY-MM-DD

## 対象レイヤー

影響を受けるレイヤー・パッケージ

## 制約の概要

1〜2文で制約を要約

## 背景

なぜこの制約が生じたか

## 制約詳細

具体的な制約内容（箇条書き）

## 影響範囲

影響を受けるファイル・モジュール

## 今後の対応

制約の解消・緩和に向けたアクション
```

## 記録すべきパターン（実例から得られた知見）

| パターン | 概要 | 例 |
|---------|------|------|
| **スケルトン→仕様準拠** | 仕様書提供に伴い、仮の型定義を正式な型に更新する判断 | 仕様確定前のスケルトン型→仕様準拠型への移行 |
| **責務の線引き** | 2つ以上のモジュール間で責務の境界を明確にする | 予算管理と計画管理の責務分割 |
| **共通コンポーネントへの集約** | 複数モジュールで重複する UI/ロジックを共通パッケージに集約する | props 統一、CSS 変数化、集約基準の決定 |
| **型の削除** | 使用箇所がゼロの型を削除する（影響範囲を明記） | 未使用ブランド型の削除 |
| **外部依存の制約** | 外部サービス・ライブラリに起因する制約の記録 | 認証基盤の制約、DB の制限事項 |

## 一覧

| No. | タイトル | 対象 | 状態 |
| --- | --- | --- | --- |
```
