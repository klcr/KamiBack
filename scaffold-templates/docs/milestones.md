```markdown
# マイルストーン（Milestones）

プロジェクトの大きな節目（マイルストーン）を定義し、完了条件を明確にするドキュメント。

## 目的

- プロジェクトの進捗を大きな区切りで把握する
- 各マイルストーンの完了条件を事前に定義し、達成を客観的に判定できるようにする
- `PROGRESS.md` のフェーズと対応づけ、進捗管理の詳細層として機能する

## PROGRESS.md との関係

`PROGRESS.md` はプロジェクト全体の **サマリー層**（状態と差分の概況）。
`docs/milestones.md` は **詳細層**（完了条件と達成基準の定義）。

- `PROGRESS.md` のフェーズIDとマイルストーン番号は一致させてもよいし、別管理でもよい
- フェーズの粒度が大きい場合、1フェーズに複数マイルストーンが対応することがある
- 進捗の概況は `PROGRESS.md` を見る。完了条件の詳細はこのファイルを見る

## テンプレート

各マイルストーンには以下を定義する:

### M{番号}: {マイルストーン名}

**対応フェーズ**: P{N}（PROGRESS.md のフェーズID）

**完了条件**:

- [ ] 条件1
- [ ] 条件2
- [ ] 条件3

**達成日**: （未達成の場合は空欄）

**備考**: （補足事項があれば記載）

## 段階導入の分解パターン

機能を段階的に導入する際の共通パターン:

```
仕様確定 → ドメイン型定義 → フロントエンドモック → API 実装 → API 接続 → 認証統合 → 結合テスト
```

各マイルストーンを上記のパターンに沿って分解すると進捗が管理しやすくなる。

**ブロックポイントの設計**: 外部サービス（認証基盤、本番 DB 等）が揃うまでは「フロントエンドモック」で止まる設計とする。ローカルストレージ等のモックで単体動作を保証し、外部サービスの準備状況に依存しない開発を可能にする。

**段階分離**: 1つのマイルストーンが外部依存を含む場合、以下の2段階に分離する:

- **Step A**: 外部サービスなしで完了できる範囲（モック・ローカル動作）
- **Step B**: 外部サービス必須の範囲（本番接続・認証統合等）

## 一覧

| No. | マイルストーン名 | 対応フェーズ | 状態 | 達成日 |
| --- | --- | --- | --- | --- |
```
