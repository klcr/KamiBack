# VG-3: 現場担当者の操作成功率

## 対応仮説

**H3: 現場の人間が迷わず撮影→確認→修正できる**

## 状態

未検証

## 前提マイルストーン

M5（レビューUI強化）

## なぜこの検証が必要か

判断基準2「現場の人間が迷わないこと」は、開発者がいくらUIを作り込んでも、実際に使う人が迷うかどうかは使ってもらわないと分からない。

エラーメッセージの日本語化やガイドUIは「迷わないはず」という開発者の推測にすぎない。この検証で推測を事実に変える。

## 検証プロトコル

> **⚠ 全ステップで人間の介入が必要。開発者以外の参加が必須。**

### 準備

- 検証参加者: **3名以上**（IT部門外の業務担当者が望ましい）
- テスト帳票: トンボ＋ページID付きで印刷済みの帳票 1枚（10フィールド）
- 帳票には活字の値が印刷済み（手書き欄がある場合は手書きで記入済み）
- スマートフォン: 参加者自身のもの（アプリインストール不要＝ブラウザアクセス）
- 検証環境: 事務机の上。特別な照明は用意しない（現場に近い条件）

### 指示の与え方（これだけ伝える）

> 「この帳票に書かれている内容をデジタルデータにしたいです。
> このURLをスマホで開いて、帳票を撮影して、結果を確認してください。」

**それ以外の説明は一切しない。** UIが自己説明的であるかを検証するため。

### 観察ポイント（検証者が記録）

参加者の操作を横で観察し、以下を記録する。口を出さないこと。

| 観察項目 | 記録内容 |
|---------|---------|
| カメラ起動まで | 何秒かかったか。迷った様子はあったか |
| 撮影 | ガイドを見て帳票を合わせられたか。何回撮り直したか |
| OCR 結果表示 | 待ち時間中の反応（不安そうか、待てているか） |
| レビュー画面の理解 | 結果が何を表しているか理解できたか |
| 修正操作 | 修正が必要なフィールドを見つけられたか。修正方法が分かったか |
| 確定操作 | 最後の「確定」ボタンを見つけて押せたか |
| エラー遭遇時 | エラーが出た場合、メッセージを読んで対処できたか |
| 完了 | 完了まで何分かかったか |
| 助けを求めた回数 | 操作中に「これどうすればいいですか？」と聞いた回数 |

### 事後インタビュー（参加者に聞く）

操作完了後、以下を聞く:

1. 難しかったところはありますか？
2. 次に何をすればいいか分からなくなった場面はありますか？
3. エラーが出たとき、何をすればいいか分かりましたか？
4. この仕組みを普段の業務で使いたいと思いますか？
5. 改善してほしい点はありますか？

## 合格基準

| 項目 | 基準 | 判定者 |
|------|------|--------|
| 完了率 | 3名中 3名が操作完了 | 人間（観察） |
| 所要時間 | 全員 5 分以内 | 人間（計測） |
| 助けを求めた回数 | 平均 1 回以下 | 人間（カウント） |
| 撮り直し回数 | 平均 2 回以下 | 人間（カウント） |
| 事後インタビュー Q4 | 3名中 2名以上が「使いたい」 | 人間（聞き取り） |

### 基準の補足

- 1名が完了できなくても、原因が特定の端末/環境に起因する場合は **条件付合格** とする
- 所要時間が 5 分を超えても、特定の工程（OCR待ち等）が原因の場合は性能問題として切り分ける

## 不合格時のアクション

| 失敗パターン | 想定原因 | 対応方針 |
|-------------|---------|---------|
| カメラ起動で迷う | カメラ権限の説明不足 | B-1 の権限要求UX改善 |
| 撮影でうまく撮れない | ガイドが不明瞭 | B-1 の撮影ガイド改善 |
| レビュー画面が理解できない | 情報量が多すぎる/少なすぎる | B-10 のレイアウト再設計 |
| 修正方法が分からない | タップターゲットが小さい/修正UIが不明瞭 | P5-4 モバイル最適化 |
| エラーメッセージが役に立たない | 技術的すぎる表現 | P5-2 エラーフィードバック改善 |
| 「使いたくない」が多数 | 全体の操作感 | UXの根本的再設計を検討（エスカレーション） |
