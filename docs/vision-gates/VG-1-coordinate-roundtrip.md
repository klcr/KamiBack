# VG-1: 座標の往復一貫性

## 対応仮説

**H1: 同一mm座標の往復再利用で、構造の事前知識だけでOCR精度が出る**

## 状態

未検証

## 前提マイルストーン

M3b（トンボ＋ページ識別コード＋印刷）+ M4a（カメラ撮影＋トンボ検出＋射影変換）

## なぜこの検証が必要か

このシステムの全体像は「mm座標で値を紙の上に配置し、同じmm座標で紙の上から値を読み取る」こと。座標が往復で一致しなければ、どれだけ OCR エンジンが優秀でも正しいフィールドを切り出せない。

ここが崩れると P4 以降の全タスクの前提が変わる。**最も早く検証すべき仮説**。

## 検証プロトコル

### Step 1: 合成画像での自動検証（AI が実施可能）

1. Module A でサンプルマニフェスト（10フィールド以上）に値をバインドして帳票画像を生成する
2. 生成画像に合成的な歪み（台形変形、回転5度）を加える
3. Module B でトンボ検出 → 射影変換 → ボックス切り出しを実行する
4. 切り出されたボックスの座標と、マニフェストの mm 座標を比較する

**自動判定基準**: 全フィールドで座標誤差 ±0.5mm 以内

### Step 2: 実プリンタ出力での実測（人間が実施）

> **⚠ 人間の介入が必要**

1. Module A で生成した帳票を **実際のプリンタで印刷する**（A4、100% スケール）
2. 印刷物のトンボ間距離を **定規で実測する**
3. マニフェストの理論値との差を記録する
4. 印刷物を **スマートフォンで撮影する**（3枚: 正対、やや斜め、照明ムラ）
5. 各撮影画像を Module B に入力し、トンボ検出 → 射影変換を実行する
6. 補正後画像のボックス位置を目視で確認する

**人間が確認すること:**

- [ ] 印刷物のトンボ間距離が理論値と ±1mm 以内か
- [ ] 「100% スケール」で印刷できたか（ブラウザの縮小/拡大が介入していないか）
- [ ] 正対撮影で全 4 トンボが検出されたか
- [ ] 斜め撮影でも 3 点以上検出されたか
- [ ] 補正後画像でボックスの位置が視覚的に正しいか（テキストがボックス内に収まっているか）

### Step 3: OCR 精度の確認（人間が判定）

> **⚠ 人間の介入が必要**

1. Step 2 の補正画像に対して OCR を実行する
2. 元の変数値と OCR 結果を対比する
3. フィールドごとの正答/誤答を記録する

**人間が判定すること:**

- [ ] 活字フィールドの正答率が 90% 以上か（95% 未満の場合は座標ずれが原因か OCR 精度が原因かを切り分ける）
- [ ] 誤答フィールドのパターン（用紙端に集中しているか、特定の文字種か）

## 合格基準

| 項目 | 基準 | 判定者 |
|------|------|--------|
| 合成画像での座標誤差 | 全フィールド ±0.5mm 以内 | 自動テスト |
| 印刷物のトンボ間距離 | 理論値と ±1mm 以内 | 人間（実測） |
| 正対撮影のトンボ検出 | 4/4 検出 | 人間（確認） |
| 補正後のボックス位置 | 目視で正しい | 人間（確認） |
| 活字 OCR 正答率 | 90% 以上 | 人間（判定） |

## 不合格時のアクション

| 失敗パターン | 想定原因 | 対応方針 |
|-------------|---------|---------|
| 印刷スケールのずれ | ブラウザの自動縮小 | 印刷ガイドの追加、または縮小率検出の仕組み |
| トンボ検出失敗 | マーク形状・サイズの問題 | トンボのデザインパラメータ調整 |
| 座標は合っているが OCR 精度が低い | 前処理・エンジンの問題 | H1 自体は否定されない。B-5/B-7 の改善で対応 |
| 用紙端のフィールドだけ精度が低い | 射影変換の端部誤差 | 端部フィールドに誤差メタデータを付与（判断基準6） |
