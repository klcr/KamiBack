# ビジョン検証ゲート（Vision Gates）

技術的な完了とは別に、**ビジョンの具体化に寄与しているか**を人間が検証する仕組み。

## なぜ必要か

テストが全パスし、lint もクリーンでも、「現場の業務担当者が手入力より速くデータ化できる」かは分からない。ビジョンは仮説であり、コードだけでは検証できない部分がある。

このフレームワークは：

1. ビジョンを **検証可能な仮説** に分解する
2. 各仮説に **人間の介入が必要な検証ポイント** を設定する
3. 検証が通るまで **次フェーズへの進行を止める**

## 3つの仮説

| ID | 仮説 | 検証のタイミング | 検証者 |
|----|------|-----------------|--------|
| H1 | 同一mm座標の往復再利用で、構造の事前知識だけでOCR精度が出る | M3b + M4a 完了後 | 開発者（実測を含む） |
| H2 | 撮影＋レビューUIで、手入力より速くデータ化できる | M4b 完了後 | 業務担当者（非IT） |
| H3 | 現場の人間が迷わず撮影→確認→修正できる | M5 完了後 | 業務担当者（非IT） |

## ゲートの運用ルール

### 1. ゲートは通過するまでブロッカーとして扱う

ゲートの状態は `PROGRESS.md` の「ビジョン検証」セクションに記録する。
`未検証` のゲートが存在するフェーズの次フェーズには進行しない。

### 2. 判定は人間が行う

AI エージェントはゲートの状態を `検証済` に変更できない。
検証結果の記録（事実の転記）は AI が行ってよいが、判定自体は人間が行う。

### 3. 不合格でも記録する

仮説が否定された場合も記録する。否定は失敗ではなく、ビジョンの修正ポイントの発見。

### 4. 検証プロトコルは事前に書く

実装を始める前に「何をどう検証するか」を決めておく。後出しで基準を変えない。

## 状態の定義

| 状態 | 意味 |
|------|------|
| 未検証 | まだ前提マイルストーンが未完了、または検証未実施 |
| 検証中 | 検証プロトコルを実施中 |
| 合格 | 人間が合格を判定した |
| 条件付合格 | 一部条件未達だが、修正方針を明記して次フェーズ進行を許可 |
| 不合格 | 仮説が否定された。ビジョンまたは実装の見直しが必要 |

## AI エージェントへの指示

- ゲートに関わるマイルストーンが完了したら、ユーザーに検証実施を促すこと
- 検証プロトコルの手順を提示し、ユーザーの判定を待つこと
- ユーザーが明示的に「合格」「不合格」を伝えるまで、次フェーズの実装に着手しないこと
- 不合格の場合、`docs/issues/reports/` に事象を起票し、対応方針をユーザーと協議すること
