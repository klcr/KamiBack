# VG-2: スループット比較（手入力 vs 撮影＋レビュー）

## 対応仮説

**H2: 撮影＋レビューUIで、手入力より速くデータ化できる**

## 状態

未検証

## 前提マイルストーン

M4b（OCR＋変数格納＋レビューUI）

## なぜこの検証が必要か

技術的に OCR が動いても、「撮影 → OCR 処理待ち → レビュー → 修正」の全体時間が手入力より遅ければ、このシステムを使う理由がない。

ビジョンの「手入力の大部分を撮影＋確認に置き換える」が実現可能かを、実際の時間計測で確認する。

## 検証プロトコル

> **⚠ 全ステップで人間の介入が必要**

### 準備

- テスト帳票: 10フィールド（氏名、日付、金額、住所等を含む標準構成）
- 帳票に手書きではなく **活字で値が印刷された** 状態（VG-1 で使用した印刷物を流用可能）
- 計測者: できれば **開発者以外** の人。開発者が行う場合はその旨を記録する
- ストップウォッチ（スマートフォンのタイマーで可）

### Step 1: 手入力のベースライン計測

1. 帳票の印刷物を横に置く
2. タイマー開始
3. 紙を見ながら 10 フィールドの値を Excel / スプレッドシートに手入力する
4. 入力完了時点でタイマー停止
5. 所要時間を記録する

**3回繰り返し**、平均値をベースラインとする。

### Step 2: 撮影＋レビューの計測

1. レビュー UI を開く
2. タイマー開始
3. スマートフォンで帳票を撮影する
4. OCR 結果が表示されるまで待つ
5. レビュー画面で各フィールドを確認し、誤りがあれば修正する
6. 全フィールド確定したらタイマー停止
7. 所要時間を記録する

**3回繰り返し**、平均値を計測値とする。

### Step 3: 内訳の記録

各回について以下の内訳を記録する:

| 工程 | 所要時間 |
|------|---------|
| 撮影（カメラ起動〜シャッター） | |
| OCR 処理待ち | |
| レビュー確認（修正なしフィールド） | |
| 修正入力（修正ありフィールド） | |
| 確定操作 | |
| **合計** | |

### Step 4: 体感の記録（人間の主観）

> **⚠ 人間の主観が重要**

以下を自由記述で記録する:

- 手入力と比べて楽に感じたか、面倒に感じたか
- 待ち時間にストレスを感じたか
- 修正操作はスムーズだったか
- 「これなら使いたい」と思えたか

## 合格基準

| 項目 | 基準 | 判定者 |
|------|------|--------|
| 全体所要時間 | 手入力ベースラインの 70% 以下 | 人間（計測） |
| OCR 処理待ち | 5 秒以内 | 人間（計測） |
| 修正が必要なフィールド数 | 10 フィールド中 2 以下 | 人間（カウント） |
| 体感評価 | 「手入力より楽」と感じる | 人間（主観） |

### 基準の補足

- 全体所要時間が 70% 以下でなくても、フィールド数 20 以上の帳票で改善が見込まれるなら **条件付合格** とする（スケーラビリティの仮説として記録）
- 体感評価が否定的でも、時間短縮が明確なら **条件付合格** とする（UX改善を P5 で対応）

## 不合格時のアクション

| 失敗パターン | 想定原因 | 対応方針 |
|-------------|---------|---------|
| OCR 処理が遅すぎる | 画像前処理・エンジン処理時間 | B-5 の最適化、並列処理の検討 |
| 修正フィールドが多すぎる | OCR 精度不足 | B-7 のエンジン改善、B-9 の閾値調整 |
| 修正操作が遅い | レビュー UI の操作性 | P5 で UI 改善 |
| 全体は速いが体感が悪い | 待ち時間の体感 | P5-3 逐次表示で改善 |
