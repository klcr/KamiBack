# ADR 002: Module A の Python/ブラウザ処理分担

## ステータス

承認

## コンテキスト

Module A（帳票出力）は、HTMLテンプレートの読み込みから印刷出力まで複数の処理ステップを持つ。これらの処理をPython側（api/）とブラウザ側（web/）のどちらで実行するかを決定する必要がある。

選択肢は3つあった:
1. **全てブラウザ側** — HTMLパースもブラウザで行い、APIはOCR処理のみ
2. **全てPython側** — サーバーサイドレンダリングでPDF等を生成
3. **Python + ブラウザの分担** — パース・検証はPython、描画・印刷はブラウザ

## 決定

**分割の境界線は「mmの世界」と「ピクセル/DOMの世界」の切れ目に置く。**

### Python側（api/）がやること — テンプレート登録時（1回だけ）

HTMLを構造化データに変換する。HTMLテンプレートはあくまで「Excel Form Editorが出力したファイル形式」であって、このシステムにとっての正規のデータはマニフェストJSONである。Pythonの仕事は、HTMLという入れ物から構造データを取り出して検証するところまで。

- HTMLパース（DOMツリーの構築）
- マニフェストJSON抽出（`<script id="template-manifest">`）
- マニフェストとDOM属性の整合性検証（`data-*-mm` と `region` の突き合わせ）
- 変数一覧・型情報の抽出
- マニフェストの拡張（`registrationMarks`, `pageIdentifier` の座標計算・追記）

### ブラウザ側（web/）がやること — 帳票印刷時（毎回）

構造化データをDOMに反映して印刷する。ブラウザは「mm座標で配置されたCSSを、物理的な印刷結果に変換する」唯一の実行環境である。

- 変数バインディング（`{{variableName}}` → 実際の値）
- トンボのDOM要素生成・挿入
- ページ識別コード（QR）の描画
- blob URL生成
- 印刷プレビュー表示
- `window.print()` 呼び出し

### 処理フロー

```
【テンプレート登録時（1回だけ）】

  HTMLテンプレート
       │
       ▼
  ┌─────────────────────────────┐
  │  Python (api/)              │
  │                             │
  │  1. HTMLパース              │
  │  2. マニフェスト抽出        │
  │  3. 整合性検証              │
  │  4. トンボ座標計算・追記    │
  │  5. ページ識別コード生成    │
  │  6. 拡張マニフェスト出力    │
  └──────────┬──────────────────┘
             │
             ▼
  拡張マニフェストJSON ← ★これが全システムの真実
  ＋ 元HTML（そのまま保持）


【帳票印刷時（毎回）】

  拡張マニフェスト ＋ 元HTML ＋ 変数値オブジェクト
       │
       ▼
  ┌─────────────────────────────┐
  │  ブラウザ (web/)            │
  │                             │
  │  1. 元HTMLをDOMに展開       │
  │  2. 変数バインディング      │
  │  3. トンボDOM要素を挿入     │
  │  4. ページ識別QRを描画      │
  │  5. 印刷プレビュー表示      │
  │  6. blob生成 → 印刷        │
  └─────────────────────────────┘
```

## 理由

### 理由1: マニフェスト検証はブラウザでやるべきではない

マニフェストの `region.x` とDOMの `data-x-mm` が食い違っている、変数名が `{{...}}` とマニフェストで不一致、といった検証はHTMLをデプロイする前に済ませておくべき工程。ブラウザ側で実行時にエラーを出しても、現場ではもう手遅れ。Python側でバリデーションエラーを返し、テンプレート作成者がその場で直せるようにする。

### 理由2: トンボ座標の計算はPython、描画はブラウザ

トンボを「用紙物理端から5mmの位置」に置く、という座標計算はPython側でやってマニフェストに書き込む。ブラウザ側はマニフェストに書かれた座標に従ってDOM要素を配置するだけ。座標の真実がマニフェストに集約されるので、設計原則「マニフェストJSONが唯一の真実」に合致する。

### 理由3: 変数バインディングはブラウザ側のほうが素直

Python側でHTMLの文字列置換をやると、HTMLの構造を壊すリスクがある（エスケープの問題、マルチバイト文字の処理など）。ブラウザならDOMの `textContent` を書き換えるだけで済み、HTMLの構造は一切壊れない。

```javascript
// ブラウザ側: DOMベースの差し替え。壊しようがない
document.querySelectorAll('.box[data-role="field"]').forEach(el => {
  const varName = el.dataset.variable;
  if (varName in values) {
    el.textContent = formatValue(values[varName], el.dataset.type);
  }
});
```

### 理由4: 印刷はブラウザの `@page` に任せるのが最も正確

CSSの `@page` ルールで用紙サイズと余白を指定し、`window.print()` で印刷する。この経路が「mm座標 → 物理的な印刷結果」の変換において最短かつ最も信頼できるパス。Python側でPDFを生成してからブラウザで開く、といった間接的な経路は変換層が増えるだけで精度が下がる。

### 理由5: ライフサイクルの違いに合致

テンプレート登録（Python側）は1回だけ走り、帳票印刷（ブラウザ側）は何度でも走る。検証やマニフェスト拡張は登録時に済ませ、印刷時はバインディングと描画だけで済む。ブラウザ側の処理が軽く保てる。

## 境界線上の判断

| 機能 | 配置 | 理由 |
|------|------|------|
| マニフェスト拡張（トンボ座標追記） | Python側 | Module Bがマニフェストだけで動くため、マニフェストにトンボ座標が入っている必要がある |
| ページ識別コード内容生成 | Python側 | `"invoice-001/0"` という文字列を決めるのはPython |
| ページ識別コードQR描画 | ブラウザ側 | canvas API や qrcode.js でレンダリング |
| `inputType` 設定 | 帳票設計時の責務 | Excel Form Editor側かマニフェストの手動編集で設定。Module AのPython側はバリデーション（未設定の検出）だけ |

## 結果

- テンプレート登録API（`POST /api/templates/parse`）がHTMLを受け取り、拡張マニフェストJSONを返す
- ブラウザ側は拡張マニフェスト＋元HTML＋変数値を入力として印刷を実行する
- 座標の真実は常にマニフェストJSONに集約される
