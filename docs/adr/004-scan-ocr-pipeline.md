# ADR 004: スキャンOCRパイプラインの設計判断

## ステータス

承認

## コンテキスト

Module B（帳票読取エンジン）の実装に着手するにあたり、パイプライン全体に影響する9つの設計判断を事前に確定する必要がある。各判断はプロジェクトの判断基準（`docs/design-principles.md`）に照らして選定した。

## 決定

### DJ-1: mm↔pixel変換の基準

**決定**: トンボ間距離から逆算する。

射影変換後の補正画像において、検出されたトンボ間のピクセル距離と、マニフェストに定義されたトンボ間のmm距離の比からスケール係数を算出する。

```
scale = tombo_distance_px / tombo_distance_mm
pixel_x = mm_x * scale
pixel_y = mm_y * scale
```

**理由**:
- 判断基準 #1「座標が嘘をつかないこと」に最も合致
- スマホ撮影画像にはDPI情報がないため、固定DPI仮定は撮影条件で破綻する
- 射影変換後の実測値を使うため、撮影距離やズームに依存しない

**却下案**:
- 固定DPI仮定（300PPI等）: 撮影条件で不正確になる
- 用紙サイズ＋画像解像度: 傾き・クロップで不正確になる

---

### DJ-2: トンボ3点検出時の4点目推定戦略

**決定**: 用紙アスペクト比＋3点から4点目を幾何推定する。ただし、推定の歪み度合いを算出し、閾値を超えた場合はエラーとする。

**アルゴリズム概要**:
1. 検出された3点から、欠落している隅を特定する
2. 用紙のアスペクト比（マニフェストの`paper.width_mm` / `paper.height_mm`）と3点の幾何関係から4点目を推定する
3. 推定結果の歪み度合い（3点から構成される角度の理想値からの偏差等）を算出する
4. 歪み度合いが閾値を超えた場合は`TomboDetectionResult.is_sufficient = False`相当のエラーを返す

**歪み度合いの判定基準**（実装時に調整）:
- 3点から構成される角度が直角から±N度以上ずれている場合はエラー
- 推定4点目と既知3点で構成される矩形の辺比がアスペクト比から±M%以上ずれている場合はエラー

**理由**:
- 判断基準 #2「現場の人間が迷わないこと」: 1隅の汚損・折れでも読取可能にする
- 判断基準 #1「座標が嘘をつかないこと」: 歪み閾値で精度保証を維持する
- 判断基準 #6「誤差は隠さず、伝える」: 推定精度を呼び出し側に伝える

**却下案**:
- 4点必須: 現場運用でリトライが増える
- 3点でアフィン変換に切替: 台形歪みが残り精度が出ない

---

### DJ-3: OCRエンジンの統合方式

**決定**: サブプロセス呼び出し。ただし、HTTPマイクロサービス（方式C）への移行を想定した設計にする。

**実装方針**:
- `OcrEngine`インターフェース（domain層、既存）は変更しない
- `SubprocessOcrEngine`（api/infrastructure層）を実装する
- サブプロセスの入出力はJSON形式で標準化する（将来HTTPに置換しやすくする）
- 環境変数でエンジンのパスや設定を注入する

**移行パス**:
```
Phase 1: SubprocessOcrEngine（ローカル開発）
Phase 2: HttpOcrEngine（本番・スケール時）
```

**理由**:
- 設計原則 #3「エンジンの差し替えを前提にする」に合致
- サブプロセスならクラッシュ耐性があり、依存の分離が容易
- JSON入出力を標準化すれば、HTTP化は薄いラッパーの追加だけで済む

**却下案**:
- Pythonバインディング: 依存が重く、venv汚染のリスク
- 初期からHTTPマイクロサービス: ローカル開発にDocker必須となり過剰

---

### DJ-4: ページ識別コードの形式

**決定**: QRコード。

**理由**:
- 検出ライブラリが豊富（pyzbar, opencv, zxing）
- 現場担当者にとって認知度が高い
- エラー訂正レベルを選択可能（Level M推奨: 15%損傷耐性）

**却下案**:
- DataMatrix: 省スペースだが検出ライブラリの選択肢が少ない

---

### DJ-5: ページ識別コード読取失敗時のフォールバック

**決定**: リトライ促進＋手動選択の組合せ。

**フロー**:
1. 1回目失敗: 「QRコードが読み取れませんでした。もう一度撮影してください」（撮影ガイド強調表示）
2. 2回目失敗: 同上（最大3回まで自動リトライ促進）
3. 3回失敗: テンプレート一覧から手動選択画面を表示

**理由**:
- 判断基準 #2「現場の人間が迷わないこと」: 段階的にガイダンスを提供
- 多くの場合は再撮影で解決するため、不要な手動操作を減らす

---

### DJ-6: OCR実行APIのレスポンス方式

**決定**: SSE（Server-Sent Events）による逐次送信。

**実装方針**:
- `POST /api/ocr/execute` → SSEストリームを返す
- イベント種別:
  - `field_result`: 1フィールドのOCR完了ごとに送信
  - `progress`: 進捗率（`{completed: N, total: M}`）
  - `complete`: 全フィールド完了時に`OcrResult`全体を送信
  - `error`: エラー発生時
- FastAPIの`StreamingResponse`で実装

**理由**:
- 判断基準 #2: フィールド数が多い帳票（20〜30）でも待ち時間なくレビュー開始可能
- SSEはHTTP上で動作するため、WebSocketより実装・インフラが単純
- フロントエンドは`EventSource` APIで受信

**注意点**:
- M4b段階ではSSEの基盤を構築し、M5.3（逐次表示UI）で完全活用する
- 一括取得も必要な場合（テスト等）は`complete`イベントのみを使用する

---

### DJ-7: inputType別の信頼度閾値

**決定**: inputType別に閾値テーブルを持つ。

**初期閾値**（実測データで調整前提）:

| inputType | 閾値 | 根拠 |
|-----------|------|------|
| `PRINTED`（活字） | 0.80 | 活字は高精度が期待できるため高めに設定 |
| `HANDWRITTEN_NUMBER`（手書き数字） | 0.60 | 手書きはばらつきが大きいため低めに設定 |
| `HANDWRITTEN_KANA`（手書き仮名） | 0.50 | 仮名は字形の個人差が最も大きい |
| `CHECKBOX`（チェックマーク） | 0.70 | 二値判定のため中程度 |

**実装方針**:
- `ocr_result_policy.py`の`determine_reading_status()`に`input_type`パラメータを追加
- 閾値テーブルはドメイン層の定数として定義（設定ファイル化はしない）
- VG-1/VG-2検証後に実測データで閾値を調整する

**理由**:
- 判断基準 #6「誤差は隠さず、伝える」: 特性に応じた閾値で適切にレビュー対象を選別

---

### DJ-8: 補正画像の保持方式

**決定**: ファイルシステム（一時ディレクトリ）。ただし、オブジェクトストレージ（S3等）への移行を想定した抽象化を行う。

**実装方針**:
- `ImageStorage`インターフェース（domain層）を定義
  - `save(image, image_id) -> str`（パスまたはURLを返す）
  - `load(image_id) -> bytes`
  - `delete(image_id) -> None`
- `LocalFileImageStorage`（api/infrastructure層）を初期実装
  - 一時ディレクトリに保存（`tempfile.mkdtemp()`ベース）
  - セッション終了時にクリーンアップ

**移行パス**:
```
Phase 1: LocalFileImageStorage（ローカル開発）
Phase 2: S3ImageStorage（本番）
```

**理由**:
- 設計原則 #3「エンジンの差し替えを前提にする」をストレージにも適用
- ローカル開発で外部依存を増やさない

---

### DJ-9: スキャンパイプラインのユースケース粒度

**決定**: 2段階に分割する（`correct_image` + `execute_ocr`）。

**ユースケース定義**:

#### UC-1: `correct_image`（画像補正）
- **入力**: 撮影画像（バイナリ）
- **処理**:
  1. ページ識別コード（QR）を検出し、テンプレートIDとページインデックスを取得
  2. トンボ検出（`TomboDetector.detect()`）
  3. 3点時は4点目推定 + 歪み判定
  4. 射影変換（`PerspectiveCorrector.correct()`）
  5. 画像前処理（グレースケール→ブラー→適応的二値化）
  6. 補正画像を保存（`ImageStorage.save()`）
- **出力**: 補正画像ID、テンプレートID、ページインデックス、トンボ検出結果（検出数、推定有無、歪み度合い）
- **API**: `POST /api/scan/correct`

#### UC-2: `execute_ocr`（OCR実行）
- **入力**: 補正画像ID、テンプレートID、ページインデックス
- **処理**:
  1. マニフェストからフィールド一覧を取得
  2. トンボ間距離からmm↔pixelスケール係数を算出（DJ-1）
  3. 各フィールドの`absoluteRegion`をピクセルに変換し、ボックスを切出
  4. `inputType`に応じたOCRエンジンで認識
  5. `ocr_result_policy.build_field_result()`で結果構築
  6. SSEで逐次送信（DJ-6）
- **出力**: `OcrResult`（SSEストリーム）
- **API**: `POST /api/scan/ocr`（SSE）

**理由**:
- M4a（画像補正）とM4b（OCR実行）のマイルストーン境界と一致
- 中間結果（補正画像）の品質をレビューUIで確認できる
- 画像補正だけ先行テスト・デバッグ可能（原則 #5「片方だけでも価値がある」）

## 影響

- `domain/src/ocr_result/ocr_result_policy.py`: `determine_reading_status()`のシグネチャ変更（`input_type`パラメータ追加）
- `domain/src/`に`ImageStorage`インターフェースを追加（配置先は要検討: `shared/`または新規`scan/`）
- `api/src/use_cases/`に`correct_image.py`と`execute_ocr.py`を追加
- `api/src/api/routes/`に`scan/`ディレクトリを追加
- `api/src/infrastructure/`にCV実装群を追加（`hough_tombo_detector.py`, `perspective_corrector.py`, `subprocess_ocr_engine.py`, `local_file_image_storage.py`）
