# 設計原則 — 帳票OCRマッピングシステム

プロジェクト全体を貫く原則。Issue/PR の起票時に「この変更はどの原則に基づくか」を明示する。

---

## ビジョン

### このシステムは何か

**紙の帳票を「構造化データの一時的な入れ物」にするシステム。**

帳票HTMLにはボックスの座標と変数名が定義されている。このシステムは、そのmm座標を使って値を紙の上に配置し（往路）、同じmm座標を使って紙の上から値を読み取る（復路）。同一の座標定義が往復で使い回される。これが設計の全体像であり、これ以上でもこれ以下でもない。

### なぜ作るのか

現場には「紙でなければ運用できない」工程がまだ大量に残っている。防水・防塵の問題、手袋をしたままの記入、複数人での回覧、監査のための原本保管。こうした理由で紙はなくならない。

一方で、紙に書かれた情報をシステムに戻す工程は、いまだに人間の手入力に依存している。このシステムは「帳票を設計した時点の構造情報を、読み取り時にもそのまま使う」ことで、手入力の大部分を撮影＋確認に置き換える。

### 何を目指さないか

**汎用OCRエンジンを作るのではない。** 任意の紙を読み取るのではなく、「自分たちが印刷した帳票」だけを読み取る。帳票の構造を事前に知っているからこそ、レイアウト認識を省略でき、座標ベースの切り出しで十分な精度が出る。

**完全自動化を目指さない。** 「8割自動、2割人間確認」が設計上の前提。OCRの誤認識は必ず起きるので、それを素早く修正できるレビューUIに注力する。完全自動化の追求は精度向上の費用対効果が急激に悪化する領域に入るため、やらない。

**紙をなくすことを目指さない。** 紙を残したまま、紙とデジタルの往復コストを下げることがゴール。

## 開発の判断基準

迷ったときに立ち返る基準を、優先度の高い順に並べる。

### 1. 座標が嘘をつかないこと

これが最優先。マニフェストJSONに書かれたmm座標が、印刷物上の物理的な位置と一致し、撮影後の補正画像上でも同じ位置を指していること。

ここがずれると全部壊れる。印刷でずれればトンボの位置が狂い、トンボが狂えば射影変換が狂い、射影変換が狂えばボックスの切り出しが狂い、全フィールドのOCR結果がおかしくなる。精度の問題はすべてこの連鎖のどこかで発生している。

**判断の実例:**
- ブラウザの印刷ダイアログで「ページに合わせて縮小」がデフォルトONの環境がある → トンボ位置がずれる原因。印刷ガイドで「100%/実寸」を案内するか、縮小率を検出して補正する仕組みが要る
- CSS の `mm` 単位がブラウザ実装によって微妙に異なる可能性 → 印刷検証フェーズで実測し、補正テーブルを持つかどうかの判断が必要

### 2. 現場の人間が迷わないこと

Module Bを使うのはIT技術者ではなく現場の業務担当者。撮影・確認・修正の各ステップで「次に何をすればいいか」が常に明示されていること。

**判断の実例:**
- トンボ検出に失敗したとき、「Error: detection failed」と出すか、「帳票の四隅が写るようにもう少し離してください」と出すか → 後者を選ぶ。エラーメッセージは技術的な原因ではなく、ユーザーが取るべき行動を伝える
- OCR処理中の待ち時間 → プログレスインジケーターではなく、「ボックスを1つ読むたびに結果を表示していく」逐次表示にする。体感待ち時間が短くなる
- 複数ページ帳票の撮影順序 → 順番を間違えても動く設計にする。ページ識別コードがあるので、どのページを撮影したかはシステム側で判定できる。「1ページ目から順番に撮ってください」という制約をユーザーに課さない

### 3. 構造が既知であるという前提を最大限に使うこと

このシステムが汎用OCRと決定的に違う点は、読み取り対象の帳票構造を事前に知っていること。この前提は設計のあらゆる場面で活用する。

**判断の実例:**
- レイアウト認識（テキストブロックの検出）を行うか → 行わない。ボックス座標はマニフェストにある。NDLOCR-LiteのDEIMv2を通す必要はなく、PARSeqの文字列認識だけを使えばよい
- OCR後のバリデーション → `variableType` が `number` なら、OCR結果に漢字が混じっていたらおかしいと即座に分かる。`date` なら日付として解釈できるかチェックできる。構造情報があるからこそ可能な後処理
- 信頼度の閾値設定 → フィールドごとに期待される文字種が分かるので、「数字フィールドに数字が入っている」場合と「文字列フィールドにひらがなが入っている」場合で、閾値を変えられる

### 4. 事前学習に依存しないこと

トンボ検出はハフ変換、歪み補正は射影変換、照明ムラは適応的二値化。すべて古典的CV手法で動く。モデルの訓練やGPUは前提としない。

この判断基準は「精度を追求しない」という意味ではない。「帳票の構造を知っている」という強い事前知識があるので、そこにMLの汎化性能を重ねる必要がない、という意味。

**判断の実例:**
- トンボの検出精度を上げるためにYOLOで学習させるか → しない。同心円＋十字マークの幾何学的特徴だけで十分な検出率が出る。学習データの収集・管理コストに見合わない
- 手書き文字認識 → これは例外。手書きは文字種の変動が大きく、古典的手法では対応しきれない。ただし対象を数字・チェックマークに限定することで、軽量なモデルで対処する

### 5. 片方のモジュールだけでも価値があること

Module A（帳票出力）だけでも「HTMLテンプレートに値を差し込んで印刷する」ツールとして成立する。Module B（帳票読取）だけでも「トンボ付き帳票をスマホで読み取る」ツールとして成立する。両方揃うと往復が閉じる。

**判断の実例:**
- Module Bの入力として「Module Aが生成した拡張マニフェスト」だけを要求するか、「元のHTMLテンプレート」も要求するか → マニフェストJSONだけで動くようにする。Module Aを使わずにマニフェストを手書きしたケースでもModule Bが動く設計
- Module AにModule B向けの設定UI（`inputType` の指定など）を組み込むか → 組み込まない。`inputType` はマニフェストJSONのフィールドとして定義するが、その値を設定するのはModule Aの責務ではない。帳票設計ツール（Excel Form Editor）側の責務

### 6. 誤差は隠さず、伝えること

座標のずれ、OCRの信頼度、画像品質の判定結果。内部で握りつぶさず、すべてユーザーまたは呼び出し元のシステムに伝える。

**判断の実例:**
- OCR結果の信頼度が0.6のフィールドを、閾値0.5だからOKとして自動確定するか → しない。信頼度0.6は「たぶん合ってるが怪しい」。スコアをそのまま返し、閾値の判断は呼び出し側またはレビューUIに委ねる
- 射影変換後の画像で、変換誤差が大きい領域（用紙端など）がある → 各フィールドの位置に応じた推定誤差をメタデータとして付与する。中心付近のフィールドは誤差が小さく、端は大きいという情報をOCR結果と一緒に返す
- ブラウザの印刷スケーリングが100%でない可能性がある → 検出できるなら警告を出す。検出できないなら「印刷後にトンボ間の距離を定規で測ってください」とガイドする

## アーキテクチャ上の原則

### マニフェストJSONが唯一の真実

往路でも復路でも、座標・変数・型の情報源はマニフェストJSONただ1つ。HTMLのDOM属性、CSS座標、OCR結果のピクセル座標は、すべてマニフェストに書かれたmm座標から導出されるか、マニフェストと照合される。

複数の場所に同じ情報を持たない。持つと必ずずれる。

### 単位変換は境界で1回だけ

mm → ピクセルの変換は、画像を受け取った直後に1回だけ行う。以降の処理はすべてピクセル座標で統一する。逆に、mm → CSSの変換はHTML生成時に1回だけ行う。変換処理がコードのあちこちに散在すると、丸め誤差の蓄積や変換方向の取り違えが起きる。

### エンジンの差し替えを前提にする

OCRエンジン（PARSeq、Tesseract、Cloud Vision APIなど）はフィールドの `inputType` に応じて切り替える設計。特定のエンジンにハードコードしない。インターフェースは「画像を渡したら文字列と信頼度が返る」に統一する。

同様に、トンボ検出も「画像を渡したら4点の座標が返る」インターフェース。将来的にハフ変換以外の手法に差し替えても、呼び出し側は変更不要。

## 判断に迷ったら

上の基準で解決しない場面では、次の問いを立てる。

**「この判断は、座標の一貫性を守るか壊すか？」** — 守るならOK。壊すなら別の方法を探す。

**「この機能は、帳票の構造が既知であるという前提を活かしているか？」** — 活かしていないなら、汎用OCRの方向に踏み出している可能性がある。引き返す。

**「これを現場の業務担当者に説明できるか？」** — 技術的に正しくても説明できない機能は、結局使われない。

## 基本原則（DDD共通）

| # | 原則 | 要点 | 違反例 |
|---|------|------|--------|
| 1 | **責務の分離** | 一つの知識を複数モジュールが重複して持たない。データの種別（事実/状態/履歴/理由）ごとに所有者を明確にする | 別コンテキストのモジュールに同じデータを二重保存する |
| 2 | **参照はするが書き戻さない** | 他コンテキストのデータはスナップショット取得のみ。元システムへの書き戻し禁止 | モジュールAからモジュールBのステータスを直接更新する |
| 3 | **段階導入と独立動作** | 各モジュールは他モジュールが未導入でも単独で動作する。連携は「あれば便利」レベルに留める | モジュールBがないとモジュールAが起動しない |
| 4 | **ワンファクト・マルチビュー** | データは発生元で1度だけ記録する。各モジュールは同じデータを異なる視座で表示する | エンティティを別コンテキストのストレージにコピーする |
| 5 | **共通識別子による横断** | 全モジュール共通の識別子（帳票ID等）を設け、横断参照の基盤とする | モジュール独自の管理番号体系を新設する |

## 並行開発ルール

複数の開発者・エージェントが同時に作業する際の制約。

| ルール | 内容 |
|--------|------|
| **パッケージ所有権** | 1パッケージを同時に2人以上が編集しない |
| **フロント+バック一体** | UI パッケージと対応する API エンドポイントは同一担当が扱う |
| **共通パッケージ変更時の義務** | 共通パッケージ（domain 等）を変更する場合、**変更前に** `docs/constraints/` に制約条件を記録する |
| **型の追加は自由** | 型の変更・削除は制約条件への記録が必須 |

## 環境変数による段階切替

外部サービスが未整備でもローカルで開発・テスト・CI が動く設計にする。

```
未設定 = ローカル動作（外部サービス不要）
```

| パターン | 切替内容 | 段階例 |
|----------|---------|--------|
| DB 実装 | インメモリ → ファイル → 本番 DB | `memory` → `jsonfile` → `production` |
| OCRエンジン | モック → NDLOCR-Lite → Cloud Vision | `mock` → `ndlocr` → `cloud` |

## コミットメッセージ規約

```
{type}({scope}): {summary}
```

| 要素 | 説明 | 例 |
|------|------|-----|
| `type` | 変更種別 | `feat`, `fix`, `docs`, `chore`, `refactor`, `test` |
| `scope` | 影響パッケージ | `domain`, `api`, `web`, `root` |
| `summary` | 変更内容の要約 | 日本語または英語 |

例:
```
feat(domain): マニフェストJSONの型定義を追加
fix(api): トンボ検出APIのレスポンス形式を修正
docs(constraints): 001-coordinate-precision を追加
chore(root): lint 設定の更新
```

## エスカレーション基準

以下に該当する場合、`docs/issues/reports/` に事象を起票してユーザーに確認する:

| 基準 | 例 |
|------|------|
| **仕様書に記載がなく推測での実装が危険** | マニフェストJSONのフィールド仕様が不明 |
| **設計原則が矛盾する状況** | 座標の一貫性と現場のユーザビリティが衝突する |
| **座標精度に影響する変更** | 単位変換ロジック、射影変換パラメータの変更 |
| **既に動作しているコードへの破壊的変更** | domain 型の削除・変更 |
| **外部システムとの連携仕様の不明点** | NDLOCR-Lite の API仕様が未確定 |
| **コスト影響が大きい変更** | DB の設計変更、クラウドリソースの追加 |
