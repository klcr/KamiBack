# 事象管理（Issues）

実装中に発生したブロッカーや課題を記録し、進捗と解決を追跡するディレクトリ。

## 目的

- 実装の詰まり（ブロッカー）を可視化し、対応状況を共有する
- 調査・対応の経緯を時系列で残し、再発時に参照できるようにする
- 解決の判定はユーザーが行う（自動解決しない）

## ファイル命名規約

`reports/{連番3桁}-{kebab-case の概要}.md`

例: `reports/001-tombo-detection-failure.md`

## 状態の定義

| 状態 | 意味 |
| --- | --- |
| 未解決 | 調査中・対応中。進捗は随時追記 |
| 解決済 | ユーザーが解決を確認した |

## テンプレート

各事象レポートには以下のセクションを含める:

```markdown
# 事象 NNN: タイトル

## 起票日

YYYY-MM-DD

## 対象レイヤー

影響を受けるパッケージ

## 事象の概要

1〜2文で要約

## 発生状況

何をしようとして、どこで詰まったか

## 調査・対応の経緯

- YYYY-MM-DD: 対応内容（時系列で追記）

## 関連ドキュメント

関連する制約条件や ADR へのリンク

## 解決

（未解決の場合は「未解決」と記載。ユーザーが解決を確認した場合に記入）

- 解決日: YYYY-MM-DD
- 解決方法: どのように解決したか
```

## エスカレーション基準

以下に該当する場合、事象レポートを起票してユーザーに確認する:

| 基準 | 例 |
|------|------|
| **仕様書に記載がなく推測での実装が危険** | マニフェストJSONのフィールド仕様が不明 |
| **設計原則が矛盾する状況** | 座標の一貫性と現場のユーザビリティが衝突する |
| **座標精度に影響する変更** | 単位変換ロジック、射影変換パラメータの変更 |
| **既に動作しているコードへの破壊的変更** | domain 型の削除・変更 |
| **外部システムとの連携仕様の不明点** | NDLOCR-Lite の API仕様が未確定 |
| **コスト影響が大きい変更** | DB の設計変更、クラウドリソースの追加 |
